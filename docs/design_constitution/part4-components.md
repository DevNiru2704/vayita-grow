# MASTER AWWWARDS WEBSITE PROMPT

## Part 4 - Component Architecture & Experience Design

> Great websites are not built section by section. They are built from a coherent system of reusable, purposeful components. Every component should belong to the same design language, interaction model, and content strategy.

---

# 1. Component Philosophy

A component is not merely a visual block.

It is a reusable experience.

Each component should have:

- A clear purpose
- A consistent visual language
- Predictable behavior
- Accessible interactions
- Responsive layouts
- Reusable architecture
- Maintainable implementation

Avoid creating one-off components unless they serve a unique storytelling moment.

---

# 2. Information Architecture First

Before designing pages, determine:

- Primary user goals
- Secondary user goals
- Business objectives
- Content hierarchy
- Conversion points
- Trust-building opportunities

Layout should emerge from information architecture-not aesthetics alone.

---

# 3. Page Structure Philosophy

Each page should feel like a narrative.

Typical progression:

1. Capture attention
2. Establish identity
3. Explain value
4. Demonstrate capability
5. Build trust
6. Address objections
7. Encourage action
8. Reinforce confidence
9. Provide next steps

Do not arrange sections randomly.

---

# 4. Hero Section

The hero establishes the emotional tone of the website.

Determine:

- Primary message
- Supporting message
- Primary CTA
- Secondary CTA
- Hero media
- Background treatment
- Motion strategy
- Initial interaction

Possible hero formats:

- Centered
- Split layout
- Editorial
- Full-screen
- Immersive
- Interactive
- Video-led
- SVG-driven
- 3D showcase
- Product-first
- Typography-first

Select based on project needs.

---

# 5. Hero Rules

The hero should answer within seconds:

- What is this?
- Why should I care?
- What should I do next?

Never sacrifice clarity for visual experimentation.

---

# 6. Navigation System

Navigation should feel effortless.

Determine:

Desktop

- Top navigation
- Floating navigation
- Transparent navigation
- Sticky navigation
- Sidebar
- Mega menu

Mobile

- Drawer
- Bottom navigation
- Expandable menu
- Minimal menu

Navigation should prioritize discoverability over creativity.

---

# 7. Navigation Behavior

Consider:

- Scroll-aware behavior
- Active section highlighting
- Smooth scrolling
- Page transitions
- Responsive adaptations
- Accessibility
- Keyboard support

Avoid navigation that surprises users.

---

# 8. Section Architecture

Each section should have a defined purpose.

Possible section types:

- Introduction
- Features
- Benefits
- Process
- Timeline
- Statistics
- Gallery
- Case studies
- Testimonials
- FAQ
- Pricing
- Team
- Blog
- Contact
- CTA
- Footer

Avoid duplicate purposes across sections.

---

# 9. Content Blocks

Content should be modular.

Examples:

Image + Text

Text + Illustration

Split Layout

Feature Grid

Editorial Columns

Timeline

Accordion

Comparison Table

Interactive Diagram

Tabbed Content

Statistics

Cards

Each block should be reusable.

---

# 10. Card System

Cards should communicate related information.

Determine:

- Interactive or static
- Elevated or flat
- Image-led or text-led
- Horizontal or vertical
- Compact or spacious

Cards should share a common design language.

---

# 11. Feature Sections

Avoid generic feature grids.

Instead determine:

Should features be presented as:

- Timeline
- Interactive cards
- Scroll story
- Diagram
- Showcase
- Comparison
- Process flow
- Visual demonstration

Choose the format that best explains the product.

---

# 12. Process Sections

When explaining workflows:

Prefer visual storytelling.

Possible layouts:

Vertical timeline

Horizontal timeline

Interactive steps

Animated path

SVG flow

Cards

Progressive reveal

Users should easily understand the journey.

---

# 13. Statistics

Numbers should reinforce credibility.

Possible presentation:

Animated counters

Charts

Infographics

Minimal metrics

Comparison bars

Circular indicators

Only animate when it enhances comprehension.

---

# 14. Testimonials

Testimonials should build trust.

Possible formats:

Cards

Carousel

Editorial quotes

Video

Case studies

Client logos

Interactive stories

Prioritize authenticity over decoration.

---

# 15. Gallery System

Determine:

Grid

Masonry

Slider

Carousel

Fullscreen

Lightbox

Scroll narrative

Hover reveal

Gallery interactions should never compromise image quality.

---

# 16. Forms

Forms should minimize friction.

Principles:

- Clear labels
- Logical grouping
- Inline validation
- Helpful feedback
- Accessible inputs
- Mobile-friendly spacing
- Keyboard support

Avoid unnecessary fields.

---

# 17. Form Feedback

Provide clear feedback for:

Focus

Hover

Validation

Submission

Success

Failure

Loading

Feedback should be immediate and understandable.

---

# 18. Call-to-Action Sections

CTAs should appear naturally within the narrative.

Avoid repetitive "Contact Us" blocks.

Instead create context-driven CTAs.

Examples:

Learn More

View Case Study

Start Free Trial

Book Demo

Explore Products

Download Guide

Request Consultation

The CTA should feel like the logical next step.

---

# 19. FAQ Sections

FAQs should reduce uncertainty.

Presentation options:

Accordion

Searchable

Categorized

Interactive

Progressive disclosure

Avoid overwhelming users with excessive content.

---

# 20. Footer

The footer is not an afterthought.

Include only relevant elements.

Possible content:

Navigation

Contact

Social links

Newsletter

Legal

Location

Quick links

Brand statement

Avoid clutter.

---

# 21. Empty States

Every empty state should communicate:

Why nothing appears

What users should do

Possible next actions

Avoid generic "No data" messages.

---

# 22. Error States

Error messages should be:

Clear

Helpful

Actionable

Friendly

Avoid technical jargon unless appropriate for the audience.

---

# 23. Loading States

Prefer:

Skeleton screens

Progress indicators

Brand animations

Optimistic UI

Avoid blank pages.

---

# 24. Search Experience

Search should support:

Autocomplete

Suggestions

Recent searches

Filtering

Keyboard navigation

Accessible controls

Search should feel instantaneous.

---

# 25. Filtering & Sorting

Determine whether users need:

Categories

Tags

Multi-select filters

Range filters

Sorting

Saved filters

Only provide controls that improve discovery.

---

# 26. Interactive Components

Potential interactive elements include:

- Accordions
- Tabs
- Sliders
- Timelines
- Before/after comparisons
- Interactive SVG diagrams
- Maps
- Product configurators
- Data visualizations

Use interaction to improve understanding-not simply to impress.

---

# 27. Storytelling Components

Award-level websites often contain bespoke storytelling moments.

Examples:

- Scroll-driven product reveals
- SVG journey maps
- Animated timelines
- Interactive ecosystems
- Layered illustrations
- Product breakdowns
- Guided explorations

Limit these to moments where they genuinely enhance the narrative.

---

# 28. Responsive Component Behavior

Components should adapt-not merely shrink.

Examples:

Desktop cards may become stacked on mobile.

Complex timelines may become vertical.

Large navigation may simplify into a drawer.

Interactive diagrams may become static illustrations.

Design each breakpoint intentionally.

---

# 29. Reusability

Every component should be evaluated for reuse.

Ask:

Can this be parameterized?

Can styling be shared?

Can behavior be abstracted?

Can future developers extend it easily?

Avoid duplication.

---

# 30. Component States

Interactive components should define all relevant states.

Examples:

Default

Hover

Focus

Active

Selected

Loading

Disabled

Success

Error

Empty

Every state should be visually distinct.

---

# 31. Accessibility

Every component should support:

Keyboard navigation

Focus management

Screen readers

ARIA where appropriate

Touch accessibility

Reduced motion

Semantic HTML

Accessibility should be built into the component-not added afterward.

---

# 32. Component Quality Checklist

Before implementing any component, verify:

- Does it solve a user problem?
- Does it belong to the design system?
- Is it reusable?
- Is it responsive?
- Is it accessible?
- Is it performant?
- Is the interaction intuitive?
- Is the content hierarchy clear?
- Does it strengthen the overall experience?

If not, redesign before implementation.

---

# 33. Final Component Philosophy

Award-level websites do not rely on extraordinary individual components.

They rely on **ordinary components executed extraordinarily well**.

Every button, card, form, image, section, and interaction should feel like part of a single, cohesive experience-refined through consistency, restraint, and craftsmanship.
